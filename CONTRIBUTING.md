# Contributing

Thank you for your interest in improving this project.

This repository presents an open framework for describing **32 scenarios of human–AI collaboration** in the creative industries.  
We welcome contributions that improve the clarity, usefulness, accessibility, and methodological quality of the framework.

## What kinds of contributions are welcome

You can contribute in several ways:

### 1. Wording improvements
Help us improve:
- scenario titles
- scenario descriptions
- questionnaire wording
- UI copy
- methodological explanations

### 2. Translation and localisation
Help us:
- fix translation issues
- improve tone and fluency
- align terminology across languages
- detect missing or inconsistent strings

### 3. Examples and case contributions
You can suggest:
- better task examples for existing scenarios
- real-world use cases
- educational applications
- edge cases that challenge the framework

### 4. Conceptual and methodological feedback
You can contribute by discussing:
- whether a scenario is clearly defined
- whether distinctions between modes are consistent
- whether the framework misses important dimensions
- how the matrix relates to real creative workflows

### 5. UI / UX improvements
You can help improve:
- navigation
- readability
- localisation behavior
- accessibility
- mobile experience
- filtering logic

### 6. Bug reports
Please report:
- broken localisation
- layout issues
- logic mismatches
- rendering issues
- scenario mapping issues

---

## Before contributing

Please check whether:
- your issue has already been reported
- your suggestion has already been discussed
- your wording change preserves the conceptual meaning of the framework

For larger conceptual changes, please open an issue first so the change can be discussed before implementation.

---

## How to contribute

### Option 1. Open an issue
Use an issue if you want to:
- report a bug
- suggest a wording change
- propose a conceptual clarification
- submit a use case
- point out a translation problem

### Option 2. Open a pull request
Use a pull request if you want to:
- fix a bug directly
- improve localisation
- revise copy
- improve documentation
- refine the interface

---

## Contribution principles

When contributing, please try to keep the following principles in mind:

### 1. Preserve conceptual consistency
This is not just a UI project.  
Changes to labels, wording, or examples may affect the conceptual logic of the framework.

### 2. Prefer clarity over jargon
The framework is intended to be useful across research, design, education, and practice.  
Please prefer clear language over unnecessarily technical language.

### 3. Distinguish concept changes from wording changes
A textual edit may look small but can change the conceptual meaning of a scenario.  
If your proposal affects the meaning rather than just the phrasing, please explain why.

### 4. Respect multilingual consistency
If you improve wording in one language, please note whether equivalent changes may also be needed in other locales.

---

## Good contribution examples

Helpful contributions include:
- “Scenario 12 sounds unnatural in French; here is a more precise wording.”
- “This scenario title is too close to another one; here is a clearer distinction.”
- “The browser locale fallback fails for zh-CN.”
- “Here is a teaching use case for scenario 19.”
- “The current description of motivation is ambiguous; here is a more precise interpretation.”

---

## Less helpful contributions

Less helpful contributions include:
- changes without explanation for conceptually important text
- large terminology rewrites with no rationale
- replacing one ambiguous term with another equally ambiguous term
- style-only changes that break consistency across languages

---

## Style guidance

### For wording
- keep the tone clear and professional
- avoid unnecessary jargon
- avoid excessive academic heaviness in UI-facing text
- preserve distinctions between automation, augmentation, and cooperation

### For translations
- prefer natural, fluent phrasing over literal translation
- preserve conceptual structure
- avoid shortening the meaning too aggressively

### For examples
- examples should clarify the scenario
- examples should not contradict the scenario logic
- examples should be concrete enough to be recognisable

---

## If you want to contribute but are unsure where to start

You can start with one of these:
- review one locale as a native speaker
- suggest a better example for one scenario
- report one confusing phrase
- test the interface on mobile
- compare two neighboring scenarios and suggest a clearer distinction

---

## Questions

If you are unsure whether something should be contributed as:
- an issue
- a pull request
- or a conceptual discussion

please open an issue first.

Thank you for helping improve the framework.
